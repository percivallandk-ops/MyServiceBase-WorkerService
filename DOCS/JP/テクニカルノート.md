## 1.はじめに

`Windows Service` を作成する際、`SCM`（`Service Control Manager` 以下`SCM`）からのイベント、 特に **`PreShutdown`** イベントのハンドリングは厄介です。
それでも`.NET Framework` では `ServiceBase` クラスを継承することで比較的容易に扱えましたが `.NET 10.0（.NET Core 系）`＋ `WorkerService` モデルでは事情が大きく異なります。扱いが非常に困難になっています。

さらに、以下のような説明が一般に流布しています：

- 「`WorkerService` でも `.UseWindowsService()` を呼べば内部で `ServiceBase` が生成される」
- 「`WorkerService` でも`ServiceBase`を継承したクラスを作ればイベントを受け取れる」

しかし、今回 `WorkerService` 用の`WidowsService` .NETライブラリを実証実験した結果、 **これらの説明は実態を正しく表していない** ことが判明しました。
全くの嘘ではないですが 、いずれの情報も **`PreShutdown`イベントを拾おうとした場合には全く機能しない** と注釈を追加しないといけません。PreShutdownイベントを拾いたいユーザーにとっては致命的な情報の欠落です。

本記事では、Windows 11 Pro 24H2 を用いて：

- `WorkerService` モデルで `ServiceBase` が実際にどう動くのか  
- `PreShutdown`/`Shutdown` を `WorkerService` で正しく受信する方法  
- 実際に実験した結果とログ
  の説明を行います。

`WorkerService` で `SCM` イベントを扱いたい方の参考になれば幸いです。

## 4.要点（結論）

- なんとか`.NET 10.0 WorkerService` モデルを壊さず、旧来の `ServiceBase` が提供していた **`OnStart`/`OnStop`/`OnShutdown`** に加え、 **`ServiceBase` には存在しない `OnPreShutdown` をハンドリングする`MyServiceBase` クラス** を実現することができた。
- *Microsoft.Extensions.Hosting.WindowsServices* パッケージに含まれる`.UseWindowsService()` を呼ぶと内部で``ServiceBase``が生成されるのは事実だが、 **ユーザーがその `ServiceBase` を操作する手段はない** 。
- `WorkerService` で`BackgroudService`クラス経由ではなく直接``ServiceBase``を継承したクラスを作ることは可能だが、このクラスを通じて`SCM`に設定はできない。つまり、このクラスでは **`PreShutdown` を受信するようには設定できない** 。
- .NET 10.0 `WorkerService` で `PreShutdown` を受信するには、 **Win32 API（`RegisterServiceCtrlHandlerEx`/`SetServiceStatus`/`ChangeServiceConfig2`）を直接叩くしかない**。  
  これは、最終手段として、やればできることは分かっていた。しかし、WindowsServiceクラスライブラリにSCMとのI/Fが用意されていないとは思えず様々なトライと実測を重ねた結果ことごとく失敗。これが唯一成功した方法となった。

## 5.現システムの動作（概要）

本システムでは、`WorkerService` モデルを崩さずに **`SCM`からの `PreShutdown`/`Shutdown` を正しく受信する** ための仕組みを実装しています。
構成のポイントは以下の通りです。

---

### 5.1. Worker クラス

`MyWorker` は `BackgroundService` を継承した**独自の `MyServiceBase` をベースクラスとして使用**します。

これにより、`WorkerService` のライフサイクル（`ExecuteAsync`/`CancellationToken`）を維持します。
一方で`SCM`からのイベントは`Winodws API`を使用して直接入手し、`ServiceBase` で実現している`OnXXXXXX`というオーバーラード関数をWorkerクラスで扱えるようにしています。このオーバーライドによりユーザーは非常に簡単なプロセスでSCMの各イベントを取り扱えるようになります。
　
### 5.2. プログラム起動時の動作

`Program.cs` ではWorkerを以下のように起動します：

- `.UseWindowsService()` を使用して `Windows Service` を登録  
- `MyWorker` を `HostedService` として `Generic Host` に登録  
- 屋台骨を生成してRUN!

```csharp
Host.CreateDefaultBuilder(args)
    .UseWindowsService()
    .ConfigureServices(services =>
    {
        services.AddHostedService<MyWorker>();
    })
    .Build()
    .Run();
```

### 5.3. SCM イベントのフック

`Worker` 起動時（`ExecuteAsync` 開始時）に **`Win32 API RegisterServiceCtrlHandlerEx` を呼び出し**  `SCM` からの制御イベントを受信するハンドラを登録します。
これにより、`Worker` で以下のイベントを直接受信できます：

- `SERVICE_CONTROL_STOP`  
- `SERVICE_CONTROL_SHUTDOWN`  
- `SERVICE_CONTROL_PRESHUTDOWN`  

### 5.4. PreShutdown の有効化

`PreShutdown` を受信するには、`SCM` に対して  **`SERVICE_ACCEPT_PRESHUTDOWN` とShutdownまでの待機時間を明示的に設定する必要があります。**
前者は `Win32 API SetServiceStatus`、後者は`Win32 API ChangeServiceConfig2`を使用して行います。

設定内容：

- `SERVICE_ACCEPT_PRESHUTDOWN` の有効化  （設定データはbitマスクなので、現在の設定を読み出してbit orを取って書き戻すのが安全です）
- `dwPreshutdownTimeout`（ミリ秒）の設定

これにより、`SCM` は `PreShutdown` を送信し、指定した時間だけshutdownを保留し、待機します。

## 5.5.`PreShutdown`処理中の `waitHint` / `checkPoint` 更新と `CancellationToken`

- `SCM` は `PreShutdown` 中に「生存確認」を要求します
- この間は`MyServiceBase` が `waitHint` / `checkPoint` を自動更新します
- システム側からの止要求が来た場合、`CancellationToken` が *true* になります
- ユーザー側の処理は待機時間内であってもこのトークンに従い、速やかに終了することが推奨されます

実証実験によると、生存確認(`waitHint` / `checkPoint` 更新)を実施しなくても、設定した待機時間は待ってくれるようです。生存確認を実施すると、待機設定時間を超えても多少は待ってくれるらしい（未確認）。`MySeriveBase`は、設定時間を超えてタスクからのリターンがない場合でも、8秒ほど粘る設計になっています。（効果のほどは未検証）。

また、設定した待機時間より明らかに短時間で処理を終える場合は、`CancellationToken`に対応しなくても支障はなかった(`CancellationToken`が発動する前に処理を終えることになるので当然だが)。しかし、`CancellationToken`はタイムアウト以外でも緊急に発生する場合があるようなので（未確認）、無視しても大丈夫というわけではないことに留意のこと。

### 5.6. 処理及び信号伝達の流れ（概要）

実際には下記説明よりもっと複雑な機構で動作しています。詳細化するとキリがないほど複雑なので、本稿では嘘にならない程度に省いて説明します。興味のある方は専門文献を参照ください。

### 5.6.1 一般的なWorkerServiceでのサービス起動と停止

### 【起動】

- `SCM` が`サービスexe` を起動し、状態が`START_PENDING`になる。

- `exe` 内で `WindowsServiceLifetime` と （`Hosting.WindowsServices`に内包されている）`ServiceBase` が初期化され、`SCM` と `.NET Host` の橋渡しが準備される。

- `SCM` が `ServiceBase`.`OnStart`のコールバックが登録されるのを待ち、`ServiceBase`->`OnStart` を呼ぶ。

- `OnStart` の中で `.NET Host` の `StartAsync` が動き出す。

- `StartAsync` が `BackgroundService.StartAsync` を呼び、（ユーザーのメイン処理）`ExecuteAsync`が非同期で動き始める。

- (ユーザーからは見えないが）`.NET Host` の `StartAsync` を起動すると`OnStart` はリターンし、 `SCM` は`RUNNING`となる

- 一方、`.NET Host` の `StartAsync`は動作を続け、ユーザーソフト(worker)のライフサイクルを監視しする。
  
  - Point1:`StartAsync`を呼ぶのは`.NET Host`であって`SCM`ではない。
  
  - Point2:`OnStart`は`SCM`が登録された関数をコールバックする。他のイベントはコントロールメッセージとして送信する。つまり、`OnStart`だけは他のメッセージとルートが異なる。
  
  - Point3:上記、とにかくもHostの`StartAsync`が開始できれば、`SCM`はサービスをスタートしたとの認識になる。

### 【停止】

- `SCM` が `ServiceBase`　に　`SERVICE_CONTROL_STOP` メッセージを送り、状態を`STOP_PENDING`にして待機する。
- `ServiceBase`が自身の`OnStop` を呼ぶ。
- `OnStop` の中で `.NET Host` の `StopAsync` が呼ばれ、内部で `StoppingToken.cancel()` が発動し、`CancelingToken`が*true*になる。
- `ExecuteAsync`（ユーザーの処理ループ）が `CancelingToken true`を検知して終了処理に入る。
- `ExecuteAsync` が完了すると `StopAsync` も完了し、`ServiceBase`.`OnStop` が停止する。
- `OnStop` が戻った時点でサービスEXEを終了する。
- `EXE`が終了し `SCM` は`STOPPED`になる。
  もし `SERVICE_CONTROL_STOP`発信から30秒以内に`EXE`が終了しなければ、`SCM` が`EXE`プロセスを強制終了する。

### 5.6.2 本システムのサービス起動と停止

### 【起動】

- `SCM` が`サービスexe` を起動し、状態が`START_PENDING`になる。

- `exe` 内で `WindowsServiceLifetime` と `ServiceBase` が初期化（`Hosting.WindowsServices`に内包されている）され、`SCM` と `.NET Host` の橋渡しが準備される。

- `SCM` が `ServiceBase`.`OnStart`のコールバックが登録されるのを待ち、`ServiceBase`.`OnStart` を呼ぶ。

- `OnStart` の中で `.NET Host` の `StartAsync` が動き始める。

- `StartAsync` が `BackgroundService.StartAsync` を呼び、（`MyServiceBase`メイン処理である）`ExecuteAsync`を非同期で呼ぶ。`ServiceBase内の`.`OnStart`は`.NET Host` の `StartAsync`を起動するとリターンする。

- この時点で`SCM`に`Start`完了が伝わり`RUNNING`となる。

- 一方、`.NET Host` の `StartAsync`は動作を続けユーザーソフト(worker)のライフサイクルを監視する。
  
  (ここまでWorkerServiceモデルで動作するので同じ動き)

- `MySeriviceBase`が`RegisterServiceCtrlHandlerEx`（Win API）を呼び出し、`SCM`との直通経路を確立、`SCM`からのコントロールをフックし、自分の`Handler`に流すよう登録する。
  （このとき`Shutdown`/`PreShutdown`のどちらをもらうか`SCM`に登録する）
  （以降`SCM`からのコントロールは`WindowsService.ServiceBase`には流れなくなる）

- `MySeriviceBase`は`Worker`の`OnStart()`を起動する。

- `OnStart()`完了後、非同期タスク`RunAsync`を起動し、ループ待機に入る。。
  
  - Point1:`SCM`からのコントロールをフックし`MyServiceBase`の`Handler`ですべてのメッセージを受け取る。代償として、`SCM`からのコントロールが`WindowsService.ServiceBase`に流れなくなる。

    ここで`SCM`のトリガーを知るのは`MyServiceBase`だけになるので、`CancelingToken`の発行や、サービス停止を`MyServiceBase`が責任をもって実施しなければならない。
  
  - Point2:`SCM`に受け取りたいイベントを直接登録するので、`PreShutdown`イベントも受信登録できる。

### 【停止】

- `SCM` が `SERVICE_CONTROL_STOP` を発信し状態が`STOP_PENDING`になる。

- `MySeriviceBase`の`Handler`が`SERVICE_CONTROL_STOP`を受信し、統括下にあるすべてのタスクに`CancellationToken=true`を送信する。

- ほぼ同時に`Worker`の`OnStop`をコールする。

- `MySeriviceBase`は、統括下にあるすべてのタスクの終了を確認し、サービス停止を`BackgroundService(Hosting)`に依頼する。

- サービスが停止する（サービスEXEが停止する）。

- `SCM`にサービスEXE停止完了が伝わり`SCM`の`STATUS`が`STOPED`になる。
  もし `SERVICE_CONTROL_STOP`発信から30秒以内にサービスが終了しなければ、`SCM` がプロセスを強制終了する。
  
  - Point1:起動や終了は.NETに任せ、`SCM`コントロールメッセージだけを独自にハンドリングする。全体としては`WorkerService`モデルを維持する。
  
  - Point2:本来それぞれハンドリングするルートが異なる`OnStart`,`OnStop`,`OnShutdown`,`OnPreShutdown`を`MySeriviceBase`で統一されたI/Fに統合し、ユーザー層(`Worker`)にハンドリング容易な形でAPIを提供する。

## 6. .NET 10.0 と .NET Framework 4.8 の違い

### ― PreShutdown 実装の観点において ―

### 6.1. .NET Framework 4.8 は「Windows 専用フレームワーク」

- `.NET Framework` は Windows 専用であり、 **`ServiceBase` を含む `Windows` との連携機能を内部に持ちます**。
- そのため、`ServiceBase` を継承してインスタンスを作成するだけで `Windows Service` (`SCM`)と直接通信できます。
- `.NET Framework`の`ServiceBase`は`SCM`に設定や状態を仲介するものでもあり、`acceptedCommands` を書き換えること（リフレクションを使う）で **`PreShutdown` を有効化することが可能** です。

---

### 6.2. .NET 10.0（.NET Core 系）は「汎用フレームワーク」

- `.NET 10.0` は  **Linux / macOS / Windows を共通の抽象化レイヤーで扱います**。

- そのため、Windows 固有の API（`ServiceBase` など）は **標準では含まれていません**。

- Windows 用の機能は「Hosting（Generic Host）」を通し、仮想OSの一つとして提供されます。

- Windowsでしか提供されない機能は仮想化の影響で様々な制約ができてしまっています。おそらくWindowsでしか提供されない機能を使えるようにすると、他のOS用にビルドしたとき動作しないトラブルが発生するからではないかと推定します。

---

### 6.3. WorkerService の本質

- `WorkerService` の `BackgroundService` は  **“`Hosting` とのインターフェース” であり、Windows との I/F ではない**。
- つまり、`WorkerService` 単体では **`SCM`（`Service Control Manager`）と直接通信する機能を持たない**。

---

### 6.4. .NET 10.0 における `ServiceBase` の扱い

- `NuGet` で `Microsoft.Extensions.Hosting.WindowsServices` を追加すると`WindowsService`が使えるようになる。`.UseWindowsService()`を実行すると内部で `ServiceBase` が生成される。
- しかし、この `ServiceBase` は  言わば`SCM`側からの一方通行。コントロールを受け取ることはできるが設定することができない。  つまり**ユーザーが`SCM`設定を変更する手段が提供されておらず、デフォルトでは未設定のPreShutdownを設定することができない。**
- リフレクションにより、`.NET Framework`  `ServiceBase`の`acceptedCommands`代替と思われる`_acceptedCommands`にアクセスはできるが、この **_acceptedCommands を変更しても `SCM` には反映されない**。

---

### 6.5. 結論（現時点での理解）

- `.NET Framework 4.8`
  → `ServiceBase` が Windows 専用 API と密結合しており、 `PreShutdown` を含む `SCM` イベントを扱える。

- `.NET 10.0 （Core）WorkerService`  
  → `ServiceBase` は内部的に存在するが、 **ユーザーが `SCM` 設定を変更する手段が提供されていない**。

- そのため、.NET 10.0 `WorkerService` で `PreShutdown` を扱うには **Win32 API を直接呼び出す方法が現時点で唯一の実用的手段** と思われる。

## 7.考察（まとめ）

- `.NET 10.0 WorkerService` には **`PreShutdown` を設定する公式 API が存在しない**  
- 筆者が確認した範囲では、 **`Win32 API`（`RegisterServiceCtrlHandlerEx` / `SetServiceStatus` / `ChangeServiceConfig2`）を直接使用する方法以外に `PreShutdown` を正しく扱える手段は見つかっていない**。
- `WorkerService` で `ServiceBase` を継承することは可能だが、 **`SCM` に設定を反映する手段が存在しないため`PreShutdown`の利用は不可**
- `.NET Framework4.8`の`ServiceBase`と`.NET 10.0`用`WorkerService`の`ServiceBase`は **名前は同じでも“別物”であり、`SCM`側から見た動作は互換でも、ユーザー側から見た互換性は無い**。
- `SCM`のハンドリングについてのみ着目すれば、`.NET Framework4.8`より`.NET 10.0（Core）`の使い勝手は下がってしまったと言える。

## A1.付録 実証実験内容と結果（概要）

### A1.1. .UseWindowsService() を起動し、

### ServiceBase を継承した MyServiceBase を Hosting に登録する

`.NET Framework`では、これでオーバーライド関数がコールされる。`WorkerService`でも同様な結果を期待。

**結果：失敗**

- My`ServiceBase` 側のコールバックは一切発生しない  
- 暫くすると例外が発生する。`OnStart`あるいは、`StartAsync`の起動で失敗と見做された模様。
- この事象の合理的解釈は以下のようになる
  - `WorkerService` の `Hosting` 内部で生成される `ServiceBase` は**ユーザーが関与できない“ブラックボックス”** である。
  - `UseWindowsService`を起動すると、コントロールメッセージ自体も伝達されない。
  - `OnXXXX`コールバックも内部で発動されて終わっている。あるいは発動しない。
- 感想としては、何のために付けた機能かと疑問を感じるレベル。

---

### A1.2. .UseWindowsService() を起動しない構成で、

### ServiceBase を継承したクラスを直接使用

**結果：部分的に成功だが本質的に失敗**

- デフォルトのイベント（`Start`/`Stop`/`Shutdown`）は受信できた
- しかし **`PreShutdown` の`acceptedCommands` を設定する方法が存在しない**  
- リフレクションで `ServiceBase` の内部を確認すると  
  `.NET Framework` の `acceptedCommands` ではなく  
  `.NET (Core)` では `_acceptedCommands` という別名フィールドが存在  
- `_acceptedCommands` に `PreShutdown` を設定しても **`SCM` に反映されず失敗
- 感想としては、開発思想が”デフォルト設定のSCMイベントは受け取れるようにした`PreShutdown`は使ってくれるな”というメッセージ性まで感じる。

---

### A1.3. .UseWindowsService() を起動し、

### BackgroundService を継承した MyServiceBase を生成し、

### Hosting 内部の ServiceBase をリフレクションで操作する試み

**結果：失敗**

- サービス起動時に、`UseWindowsService()`で`.NET 10.0（Core）`を内包してスタートするように指示した。しかし、`Hosting` が内部で保持しているはずのサービス一覧をリフレクションで列挙したが 、そもそも **`ServiceBase` が見えない** ことが判明。

- `.UseWindowsService()` が内部で生成する `ServiceBase` は動作しないわけではないことから  **`Hosting` のサービス一覧に現れない“隠しインスタンス(本来あるべきところから移動しどこか別の場所で管理されている、全く別の名前で動作しているなど)”、又は、起動時専用で“`SCM`登録後消滅するインスタンス”** であると推定される。

---

### A1.4. 最終形（本記事で公開する MyServiceBase）での実装

**結果：成功**

- `Win32 API RegisterServiceCtrlHandlerEx` を呼び出し`SCM` の制御イベントを **`WorkerService` から直接設定。**　Windows APIを使うのだからサービスモデルや`.NET`に関係なく成功することは分かっていた。

- `SetServiceStatus`,`ChangeServiceConfig2` を用いて **`SERVICE_ACCEPT_PRESHUTDOWN` の有効化** と **`PreShutdownTimeout` の設定** に成功。

- サービスの構築、起動、停止の管理は`WorkerService` で実施。`SCM`イベントのハンドリングだけを独自に実施することで`WorkerService` モデルを崩さずに **`OnStart` / `OnStop` / `OnShutdown` / `OnPreShutdown`** を何とか実現できた。

- 感想としては、横道（バイパス）を独自に作っているので、`WorkerService` のI/Fだけを用いた完全な`WorkerService`モデルではない。当然ながら他のOS用にビルドはできない。その状態で`WorkerService`モデルを維持することに意義がどれほどあるのかは少し疑問が残る。
