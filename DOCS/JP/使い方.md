## 1.利用環境の構築

注意：以下の手順はVisualStudio 2026 Communityバージョンの手順です。
　　　異なるバージョンでは手順が異なる可能性があります。

### 1.1 VisualStudioのインストール

　マイクロソフトVisualStudioをマイクロソフト社のホームページからダウンロード、インストールします。以下VisualStudio 2026 Communityバージョンを仮定します。他のバージョンでも手順やメニュー構成が多少異なるかもしれません。
　・VisualStudioのアカウント登録。
　　無料版であってもプロダクトキーとライセンス認証があります。
　　Windowsにマイクロソフトアカウントでログインしていればライセンス認証はスムーズに通ります。

### 1.2 ソリューション及びプロジェクトの作成（新規に作成を開始する場合）

- VisalStudioを起動します。
- **新規プロジェクト**を作成します。
- テンプレートの検索欄に　**C#,Windows,サービス**と条件をいれ検索すると、**言語：C#,OS:Windows,ソフトエア種別：サービス**が作成できるテンプレートが表示されます。
  　私の環境では、ここで「ワーカーサービス」を作成するか、「.NET Frameworkを使ったwindwosサービス」を作成するかの２択になります。
- WorkeServiceを作成するので**ワーカーサービス**を選択します。[次へ]
- プロジェクト名、ソリューション名、コード保管場所を指定します。[次へ]
  - フレームワークに **.NET10.0** を選択します。その他のチェックボックスは今回必要ありません。[作成]
    　(Program.cs,Worker.csのテンプレートが自動で配置されます）

### 1.3 パッケージの追加

　Windows Serviceを作成するには、**追加でWindowsService用のパッケージをインストールする**必要があります。

- メニュー[プロジェクト]->[NuGetパッケージの管理…]を選択
  　現在、インストール済みのパッケージに Microsoft.Extension.Hosting　が存在するはずです。これはWorkerSrviceを選択すると自動でインストールされます。
- [参照]タブから**WindowsServices**とタイプインして検索、*Microsoft.Extensions.Hosting.WindowsServices*を選択してインストールします。
  　これで仮想Hostを通して、Windows Servicesにアクセスする準備ができました。

### 1.4 必要ソースコードの追加

最低限４つのソースコードをプロジェクトに反映してください。workerは添付した３つのサンプルのいずれにも差し替え可能です。

## 2.MyServiceBaseの位置付け（概要）

 ソフト開発にあたり、MyServiceBaseの位置付けや挙動を知っておく必要があります。

- MyServiceBaseをHostService側からみるとWorkerに見えています。
  
  - 従って、SCMから起動がかかると、MyServiceBase内のExecuteAsyncをWorkerだと思いコールします。
  
  - 停止のときはHostService側からCancellationTokenをtureにしてきます。MyServiceBaseはループ待ちを解除してExecuteAsyncを抜け、HostServiceにサービス停止を依頼します。ここでもWorkerのように見えています。

- ユーザー作成のWorkerからはSCMと通信を橋渡しするミドルウエアに見えます。
  
  - SCMから起動、停止、Shutdown、PreShutdownがかかると、メカニズムはどうあれ、MyServiceBaseからそれぞれのオーバーライド関数がコールされます。
  
  - SCMから停止が要求されると上記OnStopがコールされると同時に、各タスクは、CancellationTokenを受け取ります。ユーザーはCancellationToken（停止要求）を検知したらできるだけ速やかにタスクを終了します。SCMはタイムアウト時間を設定しており、タイムアウトになるとSCMから強制終了されます。

- 開始時、MyServiceBaseのExecuteAsyncがコールされた後の挙動
  
  - MyServiceBaseは、ExecuteAsyncの中でOnStartをコールします。このOnStartはユーザーによりオーバーライド可能です。
  
  - OnStartからリターンを受けた後、RunAsync()を非同期で起動しtask待ち(await)になります。結果的にCancellationTokenを受け取るまでwaitすることになります。
  
  - 非同期で起動されたRunAsync()は待ちループに入り、イベント発生を待ちます。

## 3.ソフトウエアの開発（Worker.csのコーディング）

 ユーザーの仕事はWorker.csのコード作成だけです。他のソースを変更する必要はありません。（但し、Worker.csでWorkerクラス名を変更した場合は、Program.csのWorkerクラス名を変更する必要があります。）

### 3.1 Worker.cs(MyWorker.cs)のオーバーライドの作成

 （概要）
 1.MyWorker.csに用意した**ヘッダー部の定数を設定**する。
 2.要求事項に応じ、**各イベント発生に応じオーバーライド関数の中身**（＝本処理部分）を書く。
　以上です。とは言え、説明文だけではコーディングイメージが持てないと思いますので、添付サンプルの1~3のWorker.csを参照してください。
　※但し、コーディング上の注意点があります。

#### 3.1.1 定数値

- SERVICE_NAME：  サービス名を設定します。タスクマネージャやサービス管理で識別できなくなるので**他のサービスと名前が重複してはいけません。**　
- IsPreShutdown： PreShutdownイベントを受信したいときに*true*を設定。*False*の場合はShutdownイベントが発行されます。
- PreShutdownTimeout: SCMにPreShutdown処理待ち**待機時間**を設定します。原則にはこの設定時間分待機してもらえます。そして、まさにShutdownまでの時間を引きのばす目的のための唯一の手段がPreShutdownイベントの取得です。ですが、この値は、システムが持つ最大待機時間を超えることはできません。。目安30秒～40秒(30,000ms～40,000ms）。待機可能な最大時間が気になるところですがシステム環境により異なるようです。ギリギリを狙うなら適用先システムを実測して確認してください。

#### 3.1.2 オーバーライド関数

処理不要なものはコーディングする必要はありません。オーバーライドされない場合、MyServiceBase側で下表デフォルトの動作を行います。
また、それぞれ引数としてtokenが渡されます。これはSCMから停止要求が出たとき非同期で*true*に反転するcalceling tokenです。

| オーバーライド            | 起動タイミング     | 用途                     | デフォルト動作         |
| ------------------ | ----------- | ---------------------- | --------------- |
| OnStart            | 起動時         | 初期化、サブタスク起動など          | なにもしない          |
| OnStop             | 停止時         | 安全なクローズ、リソース解放など       | なにもしない          |
| OnShutdown         | Shutdown時   | 安全なクローズ、リソース解放など       | なにもしない          |
| OnPreShutdownAsync | Shutdown開始前 | 時間のかかるリソース解放、DBFLASHなど | なにもしない          |
| RunAsync           | 起動処理完了後     | イベント待ちループ              | 2s間隔cancel待ちループ |

**【重要】コーディング上の留意事項**

- OnStartその他で渡されるCancellationToken tokenは、停止要求が出たことをタスクが認識し、適切に終了処理するためのトリガーです。 
  .NET FrameworkのServiceBaseではOnStartその他関数に引数はありませんが、**MyServiceBaseはTokenを渡す**ことにしました。これはOnStartなどのオーバーライド関数から、さらに非同期で起動するサブタスクがある場合を想定し、このtokenをリレーして、停止要求を検知してもらうためです。
  　RunAsyncの引数tokenも同じものです。このtoken監視を阻害しなければ、ここにちょっとした処理を直接書くのも可です。（サンプル２参照）
- OnStartは、同期呼び出しなので、可能な限り素早く制御を戻すことが要求されます。戻さないと、Asyncループ待ちに入れず、停止要求を検知できなくなります。
  OnStartで素早く制御を戻す一つのアイディアとしては、実処理を非同期タスクとしてキックし、OnStartは遅滞なくリターンするという手法があげられます。（サンプル３参照）
- OnPreShutdownAsyncは非同期タスクとしてオーバーライドされています。これは本タスクに時間がかかっているとき、本タスクと並行して、裏で本タスク終了を待ちながら、SCMにまだ処理中であることを逐次報告する必要があるためです。
- RunAsync()はWorkerで動作させるのが定型です。必要ならWorker側でオーバーライドすることができます。
- とは言え、待ちループの中で何も処理がないのであれば、WorkerからRunAsync()を関数ごと削除しても問題ありません。この方が見通しが良くなるとも言えまず。
  （待ちループで何も処理がない例）
  Shutdown/PreShutdownだけに対応するサービス。

## 5.デバッグ

### 5.1 LOG出力

Windowsサービス、特に**シャットダウン過程でのランタイムデバッグは非常に難しい**ため、Debugビルドは実用的ではありません。
昔ながらのログ出力を後で見るデバッグが有用です。この用途にHelperでログをファイルに書き込むメソッドを用意しました。
システムがシャットダウンする直前までログを残すことができます。

- 実行の初期（例えばOnStartの最初で）で出力したいログレベルを設定します。
  
  ```cs
  　Helper.SetDebugMSGLevel(SERVICE_NAME, LogLevel);
  ```
  
  LogLevel　0:例外発生のみ/1:基本情報/2:より詳細な情報/-1:ログ出力しない
  SERVICE_NAME：Workerで定義したサービス名を渡します。複数のサービスから同時に同じログに出力する際、どのサービスからのメッセージなのかを識別します。

- コードの任意の場所でログ出力を行います。
  
  ```cs
  Helper.Write(strings, Level);
  ```
  
  stringsはログに残したい**文字列**。
  LevelはSetDebugMSGLevelのLogLevelに対応した**数値**。このlevelが設定より高い場合は出力されません。
  （例）
  
  ```cs
  portName = "COM4"
  Helper.Write($"{portName} not connected.", 1);
  ```
  
  この例は、エラー発生状況を基本情報としてログに残すもののようですね。
  
  ログは "C:\ProgramData\MyService\MyService.log" に保存され、スレッドセーフに書き込まれます。
  保存先はハードコーディングされているので、必要に応じてHelper.csの該当箇所を変更してビルドしてください。

### 5.2 タスク終了の見守り

 SCMから「停止」要求が出た時、各タスクにcalceling tokenを送りますが、何等かの原因でタスクが終了できないと、サービス（親）が終了したのにタスク（子）が生きているゾンビ状態になることがあります。しかも、なにも形跡が残らないため、何が起きてたのかユーザーに分からない場合があります。
　この状態を避けるため、**非同期処理を見守る機能**を用意しました。
 　ユーザーが見て、どのタスクか？の識別を容易にするため、あえてタスクに名前を付けて登録してもらうことにしました。
 SCMから「停止」要求が出た時、MyServiceBaseはcalceling tokenを*true*にターンし、登録された全ての非同期処理が終了するか、タイムアウトする(8秒)まで待って、サービス終了に応じます。
 タイムアウトしても該当タスクを強制終了（KILL）はしません。時間内に終了しなかったタスクの名前をログに残すに留めています。

- 見守り対象非同期タスクを登録する。(1件づつ）
  
  ```cs
  registerTask(string name, Task task): 
  ```
  
   name：タスクの名前です。ログ上での識別ですので、モジュール名でも機能名でも任意の名前を付けてください
   task：見守り対象のタスク（へのハンドル）そのものです。

- 見守り対象非同期タスクを登録する。(複数同時）
  
  ```cs
  registerTasks(params (string name, Task task)[] tasks): 
  ```
  
   (name,task)のセットを並べて登録します。  
  （指定例）
  
  ```cs
  registerTasks(("Task1", task1), ("Task2", task2), ("Task3", task3));
  ```
  
  ## 6.サービスのデプロイ方法（sc.exe ）
  
  以下は Windows 標準の `sc.exe` を使ったサービス登録手順です。  
  **SERVICE_NAME は MyWorker.cs の定義と必ず一致させてください。**
  **SCコマンド実行には管理者権限が必要です**

sc.exe コマンド サービス名 パラメーター

というコマンド構成になっています

```cmd
sc stop WorkerService2
sc delete WorkerService2
sc create WorkerService2 binPath= "FULLPATH\WorkerService2.exe" start= auto
sc description WorkerService2 "Description"
sc start WorkerService2
sc query WorkerService2
```

**binPath= ,start= の"="直後のスペースは必要です。** SCコマンドの仕様です。
コマンド詳細はマイクロソフトなど他資料を当たってください。

- sc query WorkerService2　表示例（正常に PreShutdown が有効化された場合）

```cmd
SERVICE_NAME: WorkerService2
        TYPE               : 10  WIN32_OWN_PROCESS
        STATE              : 4  RUNNING
                                (STOPPABLE, NOT_PAUSABLE, ACCEPTS_PRESHUTDOWN)
        WIN32_EXIT_CODE    : 0  (0x0)
        SERVICE_EXIT_CODE  : 0  (0x0)
        CHECKPOINT         : 0x0
        WAIT_HINT          : 0x0
```

ここで着目すべき確認ポイントは以下の項目です。

SERVICE_NAME： 意図したサービス名であること
STATE ：RUNNING であること
ACCEPTS_XXXXの記載。どのイベントを受信するのかが意図通りであること
（例）ACCEPTS_PRESHUTDOWN：PreShutdownを受信する
